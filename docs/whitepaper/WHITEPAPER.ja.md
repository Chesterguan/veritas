## 1. Vision

VERITAS は、規制された環境で動作する AIエージェントのための、軽量・決定論的・ポリシー拘束・監査可能・検証可能な実行ランタイムである。本システムは、自律性や不透明な知性よりも、信頼・制御・証拠を優先する。

VERITAS は基盤的な実行レイヤーとして設計されており、アプリケーションでも自動化製品でも重厚なガバナンスプラットフォームでもない。既存のエージェントランタイムを遅くすることなく、信頼できるものにする。

## 2. Motivation

現代の AIエージェントは強力であるが、規制された環境において根本的に信頼性が低い。その動作はしばしば非決定論的で、監査不能であり、検証が困難である。OpenClaw や ZeroClaw のようなフレームワークは、エージェントが高速・軽量・どこでもデプロイ可能であることを証明してきた——しかしそれらは、すべてのアクションがトレース可能・ポリシー制約・検証可能でなければならない環境向けには設計されていなかった。

同時に、エンタープライズガバナンスソリューションは逆の方向からこの問題に取り組んでいる。すなわち、重厚なミドルウェア・複雑なルールエンジン・低速な検証パイプラインを追加することで、軽量エージェントを有用にしていた速度とシンプルさを破壊してしまう。

VERITAS は異なる道を歩む。エージェントをゼロから作り直すのでも、官僚主義で包むのでもなく、優れたエージェントをより良く——安全・監査可能・検証可能——しつつ、すでに稼働しているランタイムの軽量・高速・コンポーザブルな性質を維持する、薄い信頼実行レイヤーを提供する。

## 3. Positioning

### VERITAS とは何か

- 信頼されたエージェント実行ランタイム
- 軽量なポリシー実施・監査レイヤー
- エージェントと現実世界の間のトラスト境界
- 規制された AIエージェントデプロイのための基盤

### VERITAS でないもの

- AI アシスタントやチャットボットではない
- 自動化プラットフォームではない
- ヘルスケアシステムや臨床ツールではない
- データプラットフォームや ETL パイプラインではない
- 重厚なガバナンスミドルウェアではない
- エージェントフレームワークの代替ではない——それらをラップするものである

### Red Hat アナロジー

Linux カーネルは高速・最小限・どこでも動作する。Red Hat はそれを置き換えたり遅くしたりしなかった——実績ある基盤の周りに信頼・サポート・認証・ガバナンスを加えることで、エンタープライズ対応にしたのである。

VERITAS は同じモデルに従う:

```
Linux Kernel        →  ZeroClaw / OpenClaw    (高速、最小限、どこでも動作)
Red Hat Enterprise  →  VERITAS                (信頼済み、ガバナンス済み、監査可能)
```

エージェント開発者はガバナンスの重さを感じるべきではない。VERITAS 上での開発は ZeroClaw 上での開発と同じように感じられるべきだ——高速・シンプル・コンポーザブル——であり、信頼はアプリケーションコードではなくランタイムによって透過的に実施される。

## 4. Design Philosophy

すべての VERITAS 設計決定を支配する 10 の原則:

1. **自律性より制御** — システムが何を行うかを決定するのは LLM ではなくシステム自身
2. **知性より証拠** — 証明可能な出力は巧妙な推論に勝る
3. **創発より決定論** — 常に予測可能な動作
4. **デフォルト拒否** — 明示的に許可されない限り何も実行されない
5. **ケイパビリティベースのセキュリティ** — 細粒度・宣言的・コンポーザブル
6. **最小トラストコンピューティングベース** — 小さなサーフェス、より少ないバグ、監査しやすい
7. **設計による監査可能性** — 後付けでなく、最初から組み込まれている
8. **検証可能な実行** — すべての出力が独立して検証できる
9. **常に人間によるオーバーライドが可能** — 機械が提案し、人間が決定する
10. **データモデル独立性** — FHIR、OMOP、またはあらゆるドメインスキーマへの依存なし

そして、すべての上に立つ一つのメタ原則:

> **確信による軽量さ。** ガバナンスはエージェントを遅く・重く・構築困難にする理由であってはならない。VERITAS が開発者体験を悪化させるなら、VERITAS は失敗している。

## 5. ZeroClaw and OpenClaw Lineage

VERITAS は、エージェントが高速・小型・実用的であることを証明した二つのオープンソースプロジェクトから実行哲学を継承している:

### ZeroClaw

ZeroClaw は Rust で書かれた超軽量 AI エージェントランタイムである。10ms 以内に起動し、約 3.4MB のバイナリとして提供され、ARM/x86/RISC-V 上で動作し、10 ドルほどの安価なハードウェアにもデプロイできる。ZeroClaw は、最小限のエージェントカーネル・明示的な実行フロー・トレイトベースのコンポーザビリティ・外部依存なし・小さなトラストコンピューティングベースを重視する。

### OpenClaw

OpenClaw は、エージェント・アズ・パーソナルアシスタントモデルをメインストリームに持ち込んだ——永続的・常時稼働・マルチチャネル（WhatsApp、Slack、Telegram 等）・セルフホスト型・100 以上の AgentSkills による拡張可能。OpenClaw は、AI エージェントがおもちゃやデモではなくインフラであることを証明した。

### VERITAS が継承するもの

ZeroClaw から:
- 最小限のカーネル設計
- 明示的な実行フロー（隠れた状態なし）
- コンポーザブルなトレイトベースアーキテクチャ
- 小さなトラストコンピューティングベース
- ポータビリティと低リソース要件

OpenClaw から:
- エージェント・アズ・インフラの考え方
- スキル/ケイパビリティによる拡張性
- 実世界のデプロイパターン
- マルチチャネル・常時稼働の運用モデル

### VERITAS が追加するもの

ZeroClaw も OpenClaw も提供するよう設計されておらず、規制された環境が要求するもの:

- **デフォルト拒否ポリシーエンジン** — すべてのアクションが実行前にポリシーチェック
- **不変監査トレイル** — 追記専用イベントストリーム、改ざん防止、リプレイ可能
- **出力検証** — 配信前のスキーマ・ルール・リスク検証
- **フォーマルトラスト境界** — LLM・ツール・データはアーキテクチャ上非信頼として扱われる
- **ケイパビリティレベルのセキュリティ** — ツールごとの権限と副作用宣言
- **人間オーバーライドフック** — 機密操作のための承認ワークフロー

VERITAS は ZeroClaw をフォークしたり置き換えたりしない。同じ軽量な基盤の上に信頼を重ねる。

## 6. System Model

VERITAS におけるエージェント実行は、制御されたケイパビリティ上で動作する決定論的ステートマシンとしてモデル化される。

### 実行ループ

```
State → Policy → Capability → Audit → Verify → Next State
```

各遷移は明示的・ポリシーチェック済み・監査済み・検証済みであり、エージェントが次の状態に進む前に完了する。ループは意図的に最小限に設計されており——隠れたミドルウェアなし・重厚なオーケストレーションなし・レイテンシを増加させる抽象化レイヤーなし。

### ステートマシンの性質

- **決定論的** — 同じ入力 + 同じポリシー = 同じ実行パス
- **観測可能** — すべての状態遷移が監査イベントとなる
- **中断可能** — 任意の遷移で人間オーバーライドが停止または変更できる
- **リプレイ可能** — 実行トレースが過去の任意の実行を再現できる

## 7. Trust Model

VERITAS における信頼は、決定論的実行・不変監査トレイル・明示的ポリシー決定・検証可能な出力から導出される。

システムは LLM の推論・外部ツール・入力データ・実行環境を本質的に信頼しない。これはパラノイアではなく、アーキテクチャ上の誠実さである。LLM はハルシネーションを起こし、ツールにはバグがあり、データは汚染される可能性があり、環境は侵害される可能性がある。

### トラスト境界

| 信頼済み | 非信頼 |
|---------|-----------|
| ランタイムコア | LLM |
| ポリシーエンジン | ツール |
| 監査エンジン | 入力データ |
| 検証器 | 外部環境 |

信頼は前提とされない。信頼は証拠から導出される——監査トレイル・ポリシーログ・検証結果。

## 8. Capability Model

ケイパビリティは、明示的なスキーマ・権限・副作用宣言を持つ制約されたツールを表す。

外部世界とのすべてのインタラクションは、ポリシー制御下のケイパビリティを通じて行われなければならない。ケイパビリティは単なる関数呼び出しではなく——以下を宣言するコントラクトである:

- 何をするか（スキーマ）
- 何を必要とするか（権限）
- 何を変更するか（副作用）
- どのようなリスクを持つか（リスクレベル）

これがケイパビリティベースのセキュリティである: エージェントはケイパビリティとして明示的に付与されていないものには一切アクセスできず、すべてのケイパビリティ呼び出しはポリシーチェックおよび監査される。

## 9. Policy and Governance

VERITAS はデフォルト拒否の実行を実施する。ポリシー決定は、サブジェクト・アクション・リソース・コンテキストを評価し、三つの結果のいずれかを決定する:

- **許可 (Allow)** — アクション実行、監査イベント記録
- **拒否 (Deny)** — アクションブロック、理由付きで監査イベント記録
- **承認要求 (Require Approval)** — 人間レビュー保留中でアクション停止

ポリシーは決定論的・説明可能・監査可能である。すべてのポリシー決定は特定のルールまでトレースでき、すべてのルールは人間の監査者が検査できる。

### 設計による軽量さ

ポリシーエンジンは重厚なビジネスルールプラットフォームではない。高速で決定論的な評価器であり——エンタープライズワークフローエンジンよりもファイアウォールに近い。ポリシー評価が追加するのはミリ秒ではなくマイクロ秒でなければならない。

## 10. Audit and Traceability

すべての実行イベントは、検証可能な実行グラフを形成する追記専用イベントストリームに記録される。各イベントは以下を含む:

- 状態遷移
- ケイパビリティ呼び出し
- ポリシー決定
- 検証結果
- タイムスタンプと因果順序

システムはリプレイ可能かつ改ざん防止の実行トレースをサポートする。任意の実行を事後に独立して再現・検証できる。

### これが重要な理由

規制された環境では、「動作した」では不十分である。*どのように*動作したか、各決定が*なぜ*下されたか、*何が*チェックされたかを証明しなければならない。監査トレイルはログファイルではない——システムが正しく動作したことを示す証拠である。

## 11. Security Model

VERITAS は最小権限・隔離されたケイパビリティ実行・厳格な境界制御を実施する。

ランタイムはシステムへの直接アクセスを許可せず、すべての外部コンポーネントを非信頼とみなす。セキュリティは追加される機能ではなく——アーキテクチャの結果である:

- デフォルト拒否ポリシー → 不正なアクションなし
- ケイパビリティベースのアクセス → アンビエント権限なし
- 不変監査トレイル → 検出されない改ざんなし
- 出力検証 → 未検証の成果物なし
- トラスト境界 → LLM またはツールへの暗黙の信頼なし

## 12. Data Model Independence

VERITAS コアランタイムは、FHIR、OMOP、HL7、またはプロプライエタリスキーマなどの特定のヘルスケアやエンタープライズデータモデルから独立している。

ドメイン固有のアダプターはケイパビリティを通じて外部に実装される。これにより VERITAS は、ヘルスケア・金融・法律・その他の規制されたドメインで動作するエージェントを、コアランタイムがドメインのデータモデルを知ることなく、ガバナンスできる。

コアはケイパビリティ・ポリシー・監査イベントを話す。ドメインは必要なものを何でも話す——それ自体がケイパビリティであるアダプターを通じて、他のすべてと同じポリシーおよび監査制御に従いながら。

## 14. Reference Domain: Healthcare

VERITAS は設計上ドメイン独立であるが、そのリファレンス実装は**ヘルスケア**を対象とする——AIエージェントデプロイにおいて最も厳しく規制され、最も高いリスクを持つ環境の一つである。

### なぜヘルスケアか

ヘルスケアは、制御されていない AIエージェントの影響が最も深刻な領域である:

- 誤った薬物相互作用チェックは患者を傷つける可能性がある
- 不正なデータアクセスは HIPAA/GDPR に違反する可能性がある
- 監査されていない臨床判断は法廷で弁護できない
- 非決定論的な出力はレビューのために再現できない

VERITAS がヘルスケアで信頼を獲得できれば、どこでも信頼を獲得できる。

### VERITAS が対処するヘルスケア固有の課題

| 課題 | VERITAS の対応 |
|---|---|
| 患者データの機密性 | ケイパビリティベースのアクセス — エージェントはポリシーが許可するものだけを見る |
| 規制コンプライアンス (HIPAA, GDPR, MDR) | 不変監査トレイルがすべてのアクセスと決定を証明する |
| 臨床意思決定支援 | 出力検証 — すべての推奨事項が配信前に検証される |
| 相互運用性 (FHIR, HL7, OMOP) | ドメインアダプターをケイパビリティとして — コアはクリーンに保たれる |
| 人間による監視要件 | 承認要求ポリシー — 機密操作に対する臨床医レビュー |
| 監査のための再現性 | リプレイ可能な実行トレース — 過去の任意のエージェント実行を再現する |

### ヘルスケアにおいて VERITAS がしないこと

- 臨床データを解釈しない
- 臨床判断を下さない
- 臨床医の判断を代替しない
- 患者記録を保存・管理しない
- FHIR/HL7 を実装しない — アダプターが行う

VERITAS はこれらのことを行う**エージェント**をガバナンスする。エージェントがポリシー内で動作し、監査可能な証拠を生成し、検証可能な出力を配信することを保証する——どの臨床システムに接続されているかにかかわらず。

## 15. Landscape and Differentiation

### エージェントフレームワーク (エージェントを構築する)

LangGraph、CrewAI、AutoGen、OpenClaw などのフレームワークは、開発者がエージェントを構築するのを助ける。オーケストレーション・ツール使用・マルチエージェント連携に焦点を当てている。これらは競合ではなく——VERITAS の潜在的な**消費者**である。任意のフレームワークで構築されたエージェントは、VERITAS ランタイム内で実行できる。

### ガードレールシステム (I/O をフィルタリングする)

Guardrails AI、LlamaFirewall、Superagent などのシステムは、入力と出力を検証する。プロンプトインジェクション・安全でないコンテンツ・スキーマ違反を捕捉する。有用ではあるが不完全——実行自体を制御することなく境界でフィルタリングする。

### エンタープライズガバナンス (ポリシー文書)

NIST AI RMF、シンガポールの Model AI Governance、AAGATE などのフレームワークは、ガバナンス原則とアセスメント方法論を提供する。*何が*起きるべきかを記述する。VERITAS はランタイムで*どのように*起きるかを実装する。

### VERITAS の位置づけ

```
┌─────────────────────────────────────────────────────┐
│              Application / Agent Code               │
│         (LangGraph, CrewAI, OpenClaw, etc.)         │
├─────────────────────────────────────────────────────┤
│                    VERITAS                           │
│   Policy Engine │ Audit Trail │ Verifier │ Caps     │
├─────────────────────────────────────────────────────┤
│              Agent Runtime Kernel                    │
│            (ZeroClaw or equivalent)                  │
└─────────────────────────────────────────────────────┘
```

VERITAS は**中間レイヤー**である——アプリケーションの下、カーネルの上。重さを加えることなく信頼を加える。

## 16. Extensibility

VERITAS は以下のための標準化されたインターフェースを提供する:

- ケイパビリティ（ドメイン固有のツールとアダプター）
- ポリシーエンジン（環境ごとのカスタムルールセット）
- 監査ストレージ（プラグ可能なバックエンド——ローカル・クラウド・ブロックチェーン）
- 検証モジュール（ドメインごとのカスタムバリデーター）

外部コントリビューターは、信頼されたコアを変更することなくシステムを拡張できる。拡張モデルは ZeroClaw のトレイトベースアーキテクチャと同じ原則に従う: コンポーザブル・スワップ可能・最小限。
