## 1. 愿景

VERITAS 是一个轻量级、确定性、策略绑定、可审计、可验证的执行运行时，专为在受监管环境中运行的 AI 智能体设计。该系统将信任、控制和证据置于自主性和不透明智能之上。

VERITAS 被设计为基础执行层——而非应用程序、自动化产品，也非重量级治理平台。它在不拖慢现有智能体运行时的前提下，使其变得可信。

## 2. 动机

现代 AI 智能体功能强大，但在受监管环境中从根本上是不可靠的。其行为往往具有非确定性、不可审计，且难以验证。OpenClaw 和 ZeroClaw 等框架已经证明，智能体可以做到快速、轻量且随处可部署——但它们并非为每个操作都必须可追溯、受策略约束且可验证的环境而设计。

与此同时，企业级治理解决方案从相反方向切入这一问题：它们增加了沉重的中间件、复杂的规则引擎和缓慢的验证流水线，彻底破坏了轻量级智能体之所以有价值的速度与简洁性。

VERITAS 走了一条不同的路。它不从头重建智能体，也不用繁文缛节将其层层包裹，而是提供一个轻薄的可信执行层，让优秀的智能体变得更好——安全、可审计、可验证——同时保留其所运行的运行时原本具备的轻量、快速和可组合特性。

## 3. 定位

### VERITAS 是什么

- 可信智能体执行运行时
- 轻量级策略执行与审计层
- 智能体与真实世界之间的信任边界
- 受监管 AI 智能体部署的基础设施

### VERITAS 不是什么

- 不是 AI 助手或聊天机器人
- 不是自动化平台
- 不是医疗系统或临床工具
- 不是数据平台或 ETL 流水线
- 不是重量级治理中间件
- 不是智能体框架的替代品——它是对框架的封装

### Red Hat 类比

Linux 内核快速、精简，随处可运行。Red Hat 并没有取代它或拖慢它——而是通过在经过验证的基础之上添加信任、支持、认证和治理，使其具备企业级就绪能力。

VERITAS 遵循同样的模式：

```
Linux Kernel        →  ZeroClaw / OpenClaw    (快速、精简、随处可运行)
Red Hat Enterprise  →  VERITAS                (可信、受治理、可审计)
```

智能体开发者不应感受到治理的负担。在 VERITAS 上构建应当如同在 ZeroClaw 上构建一样——快速、简洁、可组合——信任由运行时透明地强制执行，而非由应用代码承担。

## 4. 设计哲学

以下十条原则统领所有 VERITAS 设计决策：

1. **控制优先于自主** — 由系统而非 LLM 决定发生什么
2. **证据优先于智能** — 可证明的输出胜过巧妙的推理
3. **确定性优先于涌现** — 行为始终可预测
4. **默认拒绝** — 未经明确许可，任何操作均不得执行
5. **基于能力的安全** — 细粒度、声明式、可组合
6. **最小可信计算基** — 攻击面小、漏洞少、更易审计
7. **可审计性内建于设计** — 非事后附加，从第一天起即内置
8. **可验证执行** — 每个输出均可独立验证
9. **人工干预始终可行** — 机器提议，人类决策
10. **数据模型独立性** — 不与 FHIR、OMOP 或任何领域模式耦合

以及凌驾于一切之上的元原则：

> **轻量级是信念。** 治理不应成为智能体变得缓慢、臃肿或难以构建的原因。如果 VERITAS 使开发者体验变差，VERITAS 就是失败的。

## 5. ZeroClaw 与 OpenClaw 设计传承

VERITAS 从两个开源项目中继承其执行哲学，这两个项目已证明智能体可以做到快速、小巧且实用：

### ZeroClaw

ZeroClaw 是一个用 Rust 编写的超轻量级 AI 智能体运行时。它在 10ms 以内完成启动，以约 3.4MB 的二进制文件交付，可在 ARM/x86/RISC-V 上运行，并可部署在低至 10 美元的硬件上。ZeroClaw 强调最小化智能体内核、显式执行流、基于 trait 的可组合性、零外部依赖，以及小型可信计算基。

### OpenClaw

OpenClaw 将"智能体即个人助手"的模式带入主流——持久运行、始终在线、多渠道（WhatsApp、Slack、Telegram 等）、自托管，并可通过 100 余个 AgentSkill 进行扩展。OpenClaw 证明了 AI 智能体不是玩具或演示品，而是基础设施。

### VERITAS 的继承

继承自 ZeroClaw：
- 最小化内核设计
- 显式执行流（无隐藏状态）
- 基于 trait 的可组合架构
- 小型可信计算基
- 可移植性与低资源需求

继承自 OpenClaw：
- "智能体即基础设施"的思维方式
- 通过技能/能力实现的可扩展性
- 真实世界的部署模式
- 多渠道、始终在线的运行模式

### VERITAS 的新增能力

ZeroClaw 和 OpenClaw 均未被设计提供——而受监管环境所必需的：

- **默认拒绝策略引擎** — 每个操作在执行前均经过策略检查
- **不可变审计跟踪** — 仅追加的事件流，防篡改，可重放
- **输出验证** — 在交付前进行模式、规则和风险验证
- **正式信任边界** — LLM、工具和数据在架构上均被视为不可信
- **能力级安全** — 每个工具的权限与副作用声明
- **人工干预钩子** — 针对敏感操作的审批工作流

VERITAS 不分叉也不取代 ZeroClaw。它在同一轻量级基础之上叠加信任层。

## 6. 系统模型

VERITAS 中的智能体执行被建模为在受控能力上运行的确定性状态机。

### 执行循环

```
State → Policy → Capability → Audit → Verify → Next State
```

每次状态转换均是显式的、经过策略检查的、已审计的，并在智能体进入下一状态前完成验证。该循环有意保持精简——无隐藏中间件、无重量级编排、无增加延迟的抽象层。

### 状态机特性

- **确定性** — 相同输入 + 相同策略 = 相同执行路径
- **可观测** — 每次状态转换均为一个审计事件
- **可中断** — 人工干预可在任意转换处暂停或重定向
- **可重放** — 执行轨迹可重现任何历史运行

## 7. 信任模型

VERITAS 中的信任来源于确定性执行、不可变审计跟踪、显式策略决策和可验证输出。

系统不会固有地信任 LLM 推理、外部工具、输入数据或执行环境。这不是偏执——而是架构诚实。LLM 会产生幻觉，工具存在缺陷，数据可能被投毒，环境可能遭到入侵。

### 信任边界

| 可信 | 不可信 |
|---------|-----------|
| 运行时核心 | LLM |
| 策略引擎 | 工具 |
| 审计引擎 | 输入数据 |
| 验证器 | 外部环境 |

信任不是假设的。信任来源于证据——审计跟踪、策略日志和验证结果。

## 8. 能力模型

能力代表具有显式模式、权限和副作用声明的受约束工具。

与外部世界的所有交互必须通过处于策略控制下的能力进行。能力不仅仅是函数调用——它是一份契约，声明：

- 它做什么（模式）
- 它需要什么（权限）
- 它改变什么（副作用）
- 它带来什么风险（风险级别）

这就是基于能力的安全：智能体无法访问任何未被明确授予的能力，且每次能力调用均经过策略检查和审计。

## 9. 策略与治理

VERITAS 强制执行默认拒绝。策略决策通过评估主体、操作、资源和上下文，得出以下三种结果之一：

- **允许** — 操作继续执行，记录审计事件
- **拒绝** — 操作被阻止，记录带有原因的审计事件
- **需要审批** — 操作暂停，等待人工审核

策略是确定性的、可解释的且可审计的。每个策略决策均可追溯至特定规则，每条规则均可由人工审计员检查。

### 轻量级设计

策略引擎不是重量级的业务规则平台。它是一个快速的确定性评估器——更接近于防火墙，而非企业工作流引擎。策略评估应增加微秒级延迟，而非毫秒级延迟。

## 10. 审计与可追溯性

所有执行事件均记录在一个仅追加的事件流中，形成可验证的执行图。每个事件包含：

- 状态转换
- 能力调用
- 策略决策
- 验证结果
- 时间戳与因果排序

系统支持可重放和防篡改的执行追踪。任何执行均可在事后独立重现和验证。

### 为何重要

在受监管的环境中，"它运行了"是不够的。你必须证明*它是如何*运行的、*为何*做出了每个决策，以及*检查了什么*。审计跟踪不是日志文件——它是系统行为正确的证据。

## 11. 安全模型

VERITAS 强制执行最小权限原则、隔离的能力执行和严格的边界控制。

运行时不允许直接访问系统，并假设所有外部组件均不可信。安全性不是附加的功能——它是架构的结果：

- 默认拒绝策略 → 无未授权操作
- 基于能力的访问 → 无环境权限
- 不可变审计跟踪 → 无法检测到的篡改
- 输出验证 → 无未经验证的交付物
- 信任边界 → 对 LLM 或工具无隐式信任

## 12. 数据模型独立性

VERITAS 核心运行时独立于特定的医疗健康或企业数据模型，例如 FHIR、OMOP、HL7 或专有模式。

领域专用适配器通过能力在外部实现。这意味着 VERITAS 可以治理运行于医疗健康、金融、法律或任何其他受监管领域的智能体——而无需核心运行时了解或关心该领域的数据模型。

核心层使用能力、策略和审计事件进行通信。领域层通过适配器以其所需的任何方式进行通信——这些适配器本身就是能力，与其他所有组件一样受到相同的策略和审计控制。

## 14. 参考领域：医疗健康

尽管 VERITAS 在设计上与领域无关，但其参考实现以**医疗健康**为目标——这是 AI 智能体部署中监管最严格、风险最高的环境之一。

### 为何选择医疗健康

医疗健康是不受控 AI 智能体后果最为严重的领域：

- 错误的药物相互作用检查可能伤害患者
- 未经授权的数据访问可能违反 HIPAA/GDPR
- 未经审计的临床决策无法在法庭上得到辩护
- 非确定性输出无法重现以供审查

如果 VERITAS 能够在医疗健康领域赢得信任，它就能在任何地方赢得信任。

### VERITAS 应对的医疗健康特定挑战

| 挑战 | VERITAS 应对措施 |
|---|---|
| 患者数据敏感性 | 基于能力的访问——智能体仅能访问策略允许的内容 |
| 法规合规（HIPAA、GDPR、MDR） | 不可变审计跟踪证明每次访问和决策 |
| 临床决策支持 | 输出验证——每条建议在交付前均经过验证 |
| 互操作性（FHIR、HL7、OMOP） | 领域适配器作为能力——核心层保持整洁 |
| 人工监督要求 | 需要审批策略——针对敏感操作的临床医生审查 |
| 审计可重现性 | 可重放执行轨迹——重现任何历史智能体运行 |

### VERITAS 在医疗健康领域不做什么

- 不解读临床数据
- 不做出临床决策
- 不替代临床医生判断
- 不存储或管理患者记录
- 不实现 FHIR/HL7——适配器负责此事

VERITAS 治理的是执行这些操作的**智能体**。它确保智能体在策略范围内运行、生成可审计的证据，并交付可验证的输出——无论它连接的是哪个临床系统。

## 15. 行业格局与差异化定位

### 智能体框架（构建智能体）

LangGraph、CrewAI、AutoGen 和 OpenClaw 等框架帮助开发者构建智能体。它们专注于编排、工具使用和多智能体协调。它们不是竞争对手——而是 VERITAS 潜在的**使用方**。使用任何框架构建的智能体都可以在 VERITAS 运行时内运行。

### 护栏系统（过滤输入输出）

Guardrails AI、LlamaFirewall 和 Superagent 等系统对输入和输出进行验证。它们捕获提示注入、不安全内容和模式违规。它们有用但不完整——它们在边界处进行过滤，但不控制执行本身。

### 企业治理（策略文件）

NIST AI RMF、新加坡模型 AI 治理框架和 AAGATE 等框架提供治理原则和评估方法。它们描述*应该*发生什么。VERITAS 在运行时实现*如何*发生。

### VERITAS 所处位置

```
┌─────────────────────────────────────────────────────┐
│              Application / Agent Code               │
│         (LangGraph, CrewAI, OpenClaw, etc.)         │
├─────────────────────────────────────────────────────┤
│                    VERITAS                           │
│   Policy Engine │ Audit Trail │ Verifier │ Caps     │
├─────────────────────────────────────────────────────┤
│              Agent Runtime Kernel                    │
│            (ZeroClaw or equivalent)                  │
└─────────────────────────────────────────────────────┘
```

VERITAS 是**中间层**——位于应用层之下、内核层之上。它在不增加重量的前提下添加信任。

## 16. 可扩展性

VERITAS 为以下组件提供标准化接口：

- 能力（领域专用工具与适配器）
- 策略引擎（每个环境的自定义规则集）
- 审计存储（可插拔后端——本地、云端、区块链）
- 验证模块（每个领域的自定义验证器）

外部贡献者可以在不修改可信核心的情况下扩展系统。扩展模型遵循与 ZeroClaw 基于 trait 架构相同的原则：可组合、可替换、最小化。
